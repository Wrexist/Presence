## üõÝÔ∏è Tooling ‚Äî XcodeGen + Windows development

### Why XcodeGen
- The `.xcodeproj` is a giant, merge-conflict-prone XML blob. Committing it is painful and editing it requires Xcode (macOS only).
- `project.yml` is a single declarative YAML file ‚Äî fully editable on Windows in any text editor.
- `xcodegen generate` rebuilds the `.xcodeproj` from `project.yml` on demand (locally or in CI).
- This unlocks Windows-only development: write Swift in VS Code/Cursor, push to a branch, let GitHub Actions on macOS verify it builds.

### Gotchas
- `generateEmptyDirectories: true` is required so empty source folders don't cause warnings during generate.
- Adding a new Swift file means re-running `xcodegen generate` ‚Äî the project file does not auto-discover. CI does this every run.
- Swift Package dependencies live in `project.yml` under `packages:`, not in Xcode's Package Dependencies UI. Never edit them via Xcode (changes get blown away).
- `Info.plist` is generated by xcodegen from `info.properties` in `project.yml` because we set `GENERATE_INFOPLIST_FILE: NO` and excluded `Resources/Info.plist` from sources. To add a key, edit `project.yml`, not the generated plist.

### CI workflow
- `pr-checks.yml` runs on `macos-15` with `Xcode_26.0.app` and a Homebrew + DerivedData cache.
- Build uses `CODE_SIGNING_ALLOWED=NO` so we don't need Apple secrets just to verify a build compiles.
- Unit tests run via `xcodebuild test -only-testing:PresenceTests` against the iPhone 16 simulator.
- Failure uploads the `.xcresult` bundle as an artifact for inspection.

## üçé iOS 26 / Liquid Glass

### glassEffect API Rules (Critical)
- `.glassEffect()` requires iOS 26+ ‚Äî always gate with `#available(iOS 26.0, *)`
- `GlassEffectContainer` is required when you have multiple glass shapes that should morph together (e.g., tab bar items)
- Stacking glass layers creates visual mud ‚Äî avoid entirely
- Glass on scrollable content looks broken ‚Äî confirmed by Apple HIG
- `Reduce Transparency` accessibility setting: test every glass screen with this ON
  - Use `.background(.regularMaterial)` as fallback
- The `.glassEffect(.regular, in: RoundedRectangle(cornerRadius: GlassTokens.Radius.card))` pattern is the correct way for cards

### MapKit + Liquid Glass
- MapKit overlays in iOS 26 support glass natively via `.mapControlVisibility(.visible)`
- Custom overlays on top of the map must use glass ‚Äî confirmed pattern from Apple samples
- Map annotations (presence dots) should NOT use glass ‚Äî use custom glowing views instead

### SwiftUI Concurrency
- Swift 6 strict concurrency: all `@Observable` ViewModels must be `@MainActor`
- Location updates must come through an `actor` (LocationService as actor) to avoid data races
- WebSocket messages: receive on background actor, publish to @MainActor view models

## üåê Supabase

### PostGIS Geolocation Queries
- `ST_DWithin` is the correct function for "find users within X meters"
- Query pattern for nearby presences:
```sql
SELECT * FROM presences 
WHERE ST_DWithin(
  location, 
  ST_MakePoint($longitude, $latitude)::geography, 
  $radiusMeters
)
AND is_active = TRUE 
AND expires_at > NOW()
AND user_id != $currentUserId
LIMIT 50;
```
- Always index with `USING GIST` on geography columns
- Geography type (not geometry) for distance calculations in meters

### Real-time
- Supabase Realtime works well for presence state changes
- For high-frequency location updates, use WebSocket directly (socket.io) ‚Äî Supabase Realtime has rate limits

## ü§ñ Claude API (Opus 4.7)

### Icebreaker generation notes
- Effort level `high` is sufficient for icebreakers ‚Äî `xhigh` is overkill and slower
- Keeping system prompt < 500 tokens is important for latency
- Average latency at `high` effort: ~800ms ‚Äî acceptable with Luma connecting animation
- Always validate output: if response > 200 chars, truncate or retry once
- Edge case: if venue type is "other", the icebreaker defaults to time-based context ‚Äî works fine

### Token efficiency
- New Opus 4.7 tokenizer uses up to 35% more tokens than 4.6 ‚Äî budget accordingly
- Icebreaker calls: ~150 tokens input, ~80 tokens output ‚Üí ~0.0003 per call at standard pricing
- At 10,000 icebreakers/day ‚Üí ~$3/day in AI costs ‚Äî very manageable

## üìç Location + Privacy

### CoreLocation iOS 26
- `whenInUse` auth is sufficient for our use case ‚Äî never request `always`
- The new iOS 26 location permission dialog shows a map preview ‚Äî good for us (shows we're legit)
- Location accuracy: use `kCLLocationAccuracyHundredMeters` ‚Äî we don't need GPS precision
  - This also reduces battery impact significantly
- Geofencing: NOT used (would require `always` permission)

### Privacy UX pattern
- Show privacy explanation BEFORE requesting permission (tested: 40% better grant rate)
- The "Why we need location" screen should show Luma in `idle` state for warmth

## üé≠ Luma ‚Äî Technical Notes

### Lottie integration
- Use `LottieAnimationView` from Lottie iOS 4.x
- State switching: call `play(fromProgress:toProgress:)` for seamless state transitions
- Luma animations loop by default ‚Äî set `loopMode = .loop` for idle states
- For celebration: `loopMode = .playOnce` then transition back to idle

### Performance
- Luma at 128pt: ~2MB Lottie JSON ‚Äî acceptable
- Preload all Luma animations at app launch into `LottieAnimationCache.shared`
- On memory warning: keep idle animation, flush others
